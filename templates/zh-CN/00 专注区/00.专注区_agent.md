# 专注区 Agent

> 00 专注区/ 是用户的周工作台。此文件定义该区域的操作规范、文件生命周期、与其他区域的协作。

## 区域定义

### 用途
周工作台。当前周及上周正在进行的工作、临时文件、快速迭代的文档。

### 文件特性
- **生命周期**: 短期（通常 1-2 周）
- **修改频率**: 高
- **可见性**: 用户个人，或根据项目隔离
- **版本管理**: 不需要 git，但建议定期备份重要文件

### 生命周期
```
工作中 → 完成 → 归档（移至对应素材库）→ 删除或长期保存
```

## 操作规范

### 轻量同步（周一至周三）

执行时机：启动时自动执行

步骤：
1. 遍历 `00 专注区/` 全部文件（排除 `_归档/`），记录：
   - 文件名
   - 修改时间
   - 文件类型（文档/PPT/设计稿等）

2. 读取 `_本周.md`，对比：
   - 哪些文件在本周文档表中已记录
   - 哪些文件是新增但未记录的

3. 向用户报告新文件：
   - "看到你新加了 3 个文件：XXX、YYY、ZZZ，都是关于 [项目名] 的吗"
   - 根据文件名和内容片段做初步分类建议

4. 更新 _本周.md 的「本周文档」表

**特点**：快速、不中断工作、优先通知而不是主动整理

### 深度回溯（周四至周日）

执行时机：启动时自动执行（或用户显式触发 memory-review）

目标：确保周文件完整、专注区干净、与任务清单的对应关系清晰

步骤：

1. 全文件扫描
   ```
   ls -la 00\ 专注区/ | grep -v "^d" | grep -v "_归档"
   ```
   记录每个文件的修改时间。

2. 提取参考文件
   读取 `_本周.md` 中「任务清单」和「进展记录」已提及的文件清单。

3. 计算差集
   - **新文件**: 专注区存在、但 _本周.md 中未记录的文件
   - 标记为「待确认」或初步分类

4. 逐文件读取头部（前 30 行）
   - 理解内容类型（文档/设计稿/笔记等）
   - 识别所属项目或任务
   - 判断是否可以归档

5. 交叉比对
   - 新产出的文件对应了任务清单中的哪一项？
   - 是否有新任务产出但在清单中遗漏的？
   - 进度是否需要更新？

6. 主动更新 `_本周.md`
   - 「任务清单」: 将已完成的任务标记为 ✓
   - 「任务清单」: 如果发现新的产出，添加之前遗漏的项
   - 「进展记录」: 在对应日期下补充新文件和摘要
   - 「本周文档」: 在表中补充新产出的文件
   - 不确定的文件标注为「待确认」

7. 向用户报告
   例如：
   > "深度同步一下。发现你这周新加了 7 个文件，其中 5 个已经对应到任务清单里了。有 2 个待确认：XXX（看起来像是研究笔记？）和 YYY（新项目还是修改原有的？）。任务清单上看，'完成 Q1 规划' 已经产出对应文档了，我在进展记录里补充了。"

**特点**：深度整理、主动更新记忆、确认不确定性

## 文件管理

### 专注区内的目录结构（可选）
```
00 专注区/
├── _本周.md ← 周工作文件
├── _归档/ ← 本区域的旧文件备份
├── 项目名A/
│   ├── 核心文档
│   └── 迭代版本
└── 项目名B/
    ├── 核心文档
    └── 临时文件
```

用户可自由选择是否使用子目录。如果专注区文件很多（>20 个），建议按项目分类。

### 文件命名规范
- **日期前缀**（可选）: `2024.12.15_项目名_版本号`
- **避免**: 无意义的名字如「新建文档」、「临时」
- **清晰**: 文件名应能快速传达内容和所属任务

### 归档触发

以下情况标记为「可归档」：
- ✓ 已完成且确认不再修改的工作
- ✓ 已提交给他人或发布
- ✓ 超过 2 周未修改的文件

**归档流程**:
1. 移至对应的素材库（01 素材/、02 工具/、07 产品库/ 等）
2. 或移至 `00 专注区/_归档/`
3. 在 _本周.md 的「本周文档」表中标注「已归档」

## 与其他区域的协作

### 写入素材库（01 素材/）
- **何时**: 产出的内容有复用价值或需要长期保留
- **规范**: 遵循 01 素材/ 的 agent 规范，特别是「关于我」的敏感性规则
- **确认**: 高敏感信息需逐条确认

### 写入工具库（02 工具/）
- **何时**: 产出了可复用的模板、提示词、框架
- **规范**: 添加适当的元数据标签，方便后续检索
- **版本**: 记录初始版本和演进

### 写入产品库（07 产品库/）
- **何时**: 涉及产品设计、研究、竞品分析的工作
- **规范**: 遵循产品区域的版本管理和审查流程

### 与记忆系统的同步
- **_本周.md**: 专注区的「进展记录」自动喂入 MEMORY_LOG
- **MEMORY.md**: 识别新的「工作模式」，供周复盘时确认
- **USER.md**: 稳定的工作习惯模式（★★★ 条目）提炼到此

## 常见操作

### 快速更新 _本周.md
用户直接编辑 _本周.md，AI 在深度回溯时检测并同步。

### 问 AI 「这个文件应该归档吗」
AI 根据文件修改时间、完成度、与任务的对应关系给出建议。

### 问 AI 「本周的工作进度怎样」
AI 基于专注区扫描 + _本周.md 的差集，给出客观的进度评估。

### 为某个项目清理专注区
- 提供项目名
- AI 扫描相关文件
- 分类：还在进行 / 可以归档 / 需要确认
- 用户确认后批量移动

## 敏感内容处理

如果专注区包含高敏感信息（如客户名单、财务数据、未公开的产品信息）：
- 在 _本周.md 中用「待确认」标记，不展开内容
- 更新记忆时避免具体信息，只记录「处理了敏感数据」
- 必要时向用户逐条确认信息的可见范围

## 区域代理规则冲突解决

如果本规范与其他区域的规则冲突（如归档时涉及 01 素材的敏感规则）：
- **优先级**: 敏感性更高的规则优先
- **原则**: 更严格的规范生效
- 例：归档到 01 素材 时，必须遵循「01 素材」的 agent 规范
