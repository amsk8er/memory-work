# Materials Library Agent

## Zone Purpose

The **Materials library** serves as your personal archive of knowledge, methodology, and reference materials. It contains:
- Personal identity documents and profiles
- Methodology frameworks and thinking patterns
- Reading notes and knowledge captures
- Writing style guides and voice documentation
- Contact records and relationship notes
- Archived conversations and interaction logs

## Sensitivity Rules by Directory

### About Me/ â€” ðŸ”’ HIGH SENSITIVITY

**Rule: Never modify existing content proactively.**

- Any changes to existing files require **item-by-item confirmation**
- Do not infer or auto-update based on observations
- Always propose changes as suggestions, never execute
- Example: "I noticed you said X in Profile.md, but Y contradicts it. Should I update?"
- New additions: OK if directly from user ("add this to About Me")
- Archiving old versions: Propose before moving to `_archive/`

## Filing Decision Logic

**File by retrieval context, not source.**

When deciding where to file something:
1. Ask: "Where would I look for this next time I need it?"
2. Don't file by "who told me" or "what format it arrived in"
3. Examples:
   - Email with feedback â†’ People/Feedback Archive, not Inbox
   - Screenshot of a technique â†’ Thinking/, not Downloads
   - Reading note from X person â†’ Reading/, not People/
   - Pattern learned from experience â†’ Methodology/

## Daily Workflow Rules

- **Capture**: File materials as encountered
- **Organize**: Weekly light reorganization OK
- **Protect**: HIGH sensitivity areas use confirmation protocol
- **Archive**: Move outdated material to `_archive/` with timestamp
- **Review**: Quarterly review of About Me/ to ensure accuracy

## Agent Behavior in This Zone

When working in Materials:
- Propose HIGH sensitivity changes (don't execute)
- Execute LOW sensitivity changes (auto-organize, auto-file)
- Confirm MEDIUM sensitivity changes (explain rationale, wait for approval)
- Use retrieval-based filing logic
- Preserve document history (don't overwrite; use versions)
- Never delete content; archive instead
